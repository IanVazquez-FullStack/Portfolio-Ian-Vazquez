# ¿Por qué pnpm es el rey indiscutido de Node.js en 2026? (Y el peligro que corre npm)

Si sigues usando `npm i` por pura inercia, estás perdiendo tiempo, espacio en disco y, lo más preocupante, estás exponiendo tus proyectos a riesgos de seguridad que ya le han costado millones a la industria.

Hoy en día, **pnpm** se ha consolidado no solo como la alternativa más rápida, sino como el estándar para proyectos modernos y monorepos. Vamos a desglosar por qué deberías migrar hoy mismo, cómo funciona su magia bajo el capó con ejemplos de código, y la cruda realidad sobre la seguridad en el ecosistema de JavaScript.

---

## 1. El gran problema de npm: Duplicación y lentitud

Cada vez que corres `npm install`, npm descarga los paquetes y los copia físicamente en la carpeta `node_modules` de tu proyecto. Si tienes diez proyectos que usan React, tienes diez copias idénticas de React inundando tu disco duro.

### La solución de pnpm: El Almacenamiento Global (Content-Addressable Store)

pnpm funciona de forma radicalmente distinta. Guarda todos los paquetes en un único lugar de tu disco rígido (`~/.local/share/pnpm/store` o similar). Luego, en la carpeta `node_modules` de tu proyecto, crea **enlaces duros (hard links)** y **enlaces simbólicos (symlinks)** hacia ese almacén central.

Si diez proyectos usan la misma versión de una librería, **físicamente solo ocupa espacio una vez**.

### Comparativa visual de estructura

Con **npm**, la estructura es redundante y pesada:

```text
mi-proyecto/
└── node_modules/
    ├── lodash/ (Copia física de 5MB)
    └── express/ (Copia física de 10MB)
```

Con **pnpm**, es una red de punteros ultra ligera:

```text
mi-proyecto/
└── node_modules/
    ├── lodash -> ~/.pnpm-store/v3/files/ab/cd123... (Enlace)
    └── express -> ~/.pnpm-store/v3/files/ef/gh456... (Enlace)
```

## 2. Código en acción: Eficiencia y Monorepos

Instalar pnpm es ridículamente sencillo. Si ya tienes Node.js, viene integrado gracias a `corepack`:

```bash
# Activar corepack e instalar pnpm globalmente
corepack enable
corepack prepare pnpm@latest --activate
```

### Comandos del día a día (npm vs pnpm)

La transición es casi transparente porque los comandos son prácticamente idénticos:

```bash
# npm
npm install express
npm run dev
npm uninstall lodash

# pnpm
pnpm add express
pnpm dev             # ¡No hace falta escribir "run"!
pnpm remove lodash
```

### Ventaja letal: Gestión de Monorepos (Workspaces)

Si trabajas en un proyecto grande con múltiples paquetes (un frontend, un backend y componentes compartidos), npm se vuelve una pesadilla de configuración. pnpm maneja esto de forma nativa con un archivo `pnpm-workspace.yaml`:

```yaml
# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'
```

Para instalar una dependencia específica en un solo sub-proyecto desde la raíz, usas el flag `--filter`:

```bash
# Instala jest solo en el paquete de utilidades
pnpm --filter ./packages/utils add -D jest
```

## 3. Seguridad: Las "Dependencias Fantasma"

Uno de los peligros más silenciosos de npm es el **aplanamiento de la estructura de node_modules** (hoisting).

Cuando instalas un paquete $A$ con npm, y ese paquete depende internamente de un paquete $B$, npm sube el paquete $B$ a la raíz de tu `node_modules`. Esto significa que tu código puede hacer un `require('B')` o `import B from 'B'` **sin haberlo declarado en tu `package.json`**.

### ¿Por qué esto es un peligro?

Si el autor del paquete $A$ decide eliminar el paquete $B$ en una actualización, tu aplicación se romperá de golpe en producción porque tu código dependía de un "fantasma".

### Cómo lo resuelve pnpm

pnpm crea una estructura estrictamente no flotante. **Tu código solo puede acceder a lo que declaraste explícitamente en tu `package.json`**.

Si intentas hacer trampa, pnpm te frena en seco:

```javascript
// index.js
// Supongamos que instalamos 'express', y este usa 'mime' internamente.

import express from 'express'; // OK: Declarado en package.json
import mime from 'mime';       // ¡ERROR en pnpm! No está en tu package.json
```

Al ejecutarlo con pnpm, el entorno lanzará un error de módulo no encontrado (`ERR_MODULE_NOT_FOUND`), protegiéndote de arquitecturas frágiles.

## 4. El lado oscuro: Ataques a la cadena de suministro en npm

El ecosistema de Node.js es el más grande del mundo, y eso lo convierte en el blanco favorito de los cibercriminales. Los ataques a la **cadena de suministro (Supply Chain Attacks)** han crecido exponencialmente en los últimos años, y la estructura permisiva de npm suele facilitar su propagación.

### Tipos de grandes ataques que azotan a npm

1. **Typosquatting (Suplantación de identidad por errores de tipeo):**
Los atacantes publican paquetes maliciosos con nombres muy similares a los populares. Por ejemplo, en lugar de `lodash`, publican `lodas` o `lo-dash`. Si un desarrollador se equivoca al tipear `npm i lodas`, instala un malware que puede robar variables de entorno (`.env`) y tokens de acceso de AWS o Git.

2. **Secuestro de cuentas de mantenedores (Account Takeover):**
Hackers logran vulnerar la cuenta de un desarrollador legítimo de una librería usada por millones (mediante phishing o filtración de contraseñas). Una vez dentro, suben una versión maliciosa (ej. `v2.4.1`) que inyecta código de minería de criptomonedas o scripts de robo de tarjetas de crédito. Un caso histórico muy famoso fue el del paquete `ua-parser-js`.

3. **Confusión de Dependencias (Dependency Confusion):**
Muchas empresas usan paquetes internos privados (ejemplo: `@mi-empresa/auth`). Si un atacante registra ese mismo nombre exacto de forma pública en el registro oficial de npm con una versión altísima (ej. `v99.9.9`), el comando `npm install` de la empresa, por defecto, priorizará el paquete público malicioso sobre el privado.

---

## 5. ¿Cómo nos protege pnpm ante este caos?

Aunque pnpm sigue descargando los paquetes desde el mismo registro de npm, su arquitectura mitiga de forma drástica el impacto de estos ataques:

* **Inmutabilidad del almacenamiento global:** Si un paquete malicioso intenta modificar los archivos dentro de `node_modules` en tiempo de ejecución, pnpm lo detecta. Como los archivos están enlazados al almacenamiento global, pnpm puede verificar los hashes criptográficos. Si un archivo fue alterado, se invalida.
* **Bloqueo estricto de scripts de instalación:** Muchos malwares se ejecutan en el momento exacto de la instalación usando los scripts `preinstall` o `postinstall`. pnpm te permite desactivar la ejecución de scripts de paquetes de terceros de forma mucho más granular y segura con la opción `--ignore-scripts` o configurando paranoicamente el archivo `.npmrc`.
* **Verificación de firmas de contenido:** El archivo `pnpm-lock.yaml` almacena el hash exacto (`integrity`) de cada archivo. pnpm es sumamente estricto: si el contenido descargado no coincide al 100% con el hash registrado, la instalación falla inmediatamente, deteniendo ataques donde el código del paquete fue cambiado silenciosamente en el servidor.

Ejemplo de la rigidez de un `pnpm-lock.yaml`:

```yaml
/express/4.18.2:
  resolution: {integrity: sha512-5/iE6XBExTJIxsnfKpGgIeiA4oeZ+h3wcAj7vAyCqOhFa6bCca4M2E/uiP2wOPn5eA==}
  engines: {node: '>= 0.10.0'}
  dependencies:
    accepts: 1.3.8
    # ... cada subdependencia está perfectamente blindada con su hash
```

## Conclusión: La hora de migrar es ahora

Seguir usando npm en proyectos nuevos es mantener una deuda técnica técnica y de seguridad innecesaria.

| Característica | npm | pnpm |
| --- | --- | --- |
| Velocidad | Lenta (re-descarga todo) | Ultra rápida (usa caché global) |
| Espacio en disco | Gigantesco (duplica archivos) | Mínimo (enlaces duros) |
| Estructura | Plana (permite dependencias fantasma) | Estricta (seguridad por diseño) |
| Monorepos | Requiere herramientas extra / complejo | Soporte nativo excelente |

Pasarte a pnpm no requiere que reescribas tu código. Solo borra tu `node_modules`, borra el archivo `package-lock.json`, y ejecuta `pnpm install`. Tu disco duro, tu paciencia y el equipo de seguridad de tu empresa te lo van a agradecer.
